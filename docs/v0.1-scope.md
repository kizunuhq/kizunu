# v0.1 Scope

Initial version focused on running a real end-to-end pilot. Every decision here is justified by the reference pilot use case below. Nothing decorative.

## Reference pilot use case

A small BDR team running outbound on WhatsApp, with Pipedrive as the system of record:

- BDRs move deals in Pipedrive into a specific column (e.g., "In Follow-up Level 1").
- The system captures the event and starts a cadence of neutral WhatsApp messages.
- Each BDR has their own WhatsApp number (one Evolution instance per BDR), paid by the company for a personalized touch.
- Every executed touch is logged as an Activity on the corresponding Pipedrive deal.
- Any reply from the lead immediately pauses the cadence; the deal moves to "Replied/Scheduling".
- Exhaustion without a reply marks the deal as `lost` with reason "No Reply - Follow-up L1".

This use case is the v0.1 contract. If any part of it doesn't run, the scope isn't done.

## Core concepts

### Cadence is the aggregate

Cadence is a first-class entity, not a composition of workflow nodes. Closed vocabulary: entry trigger, sequence of steps, stop policy, named exit hooks (`onReply`, `onExhausted`, `onComplete`). No arbitrary conditional branching. No workflow engine.

A workflow builder is entirely out of scope for v0.1. Cadence preserves the aggregate and the ubiquitous language of sales.

### Channels are configured instances, not singletons

Each `ChannelAccount` is a plugin instance with its own credentials, owned by the workspace. BDRs gain access via `ChannelAccess` (explicit sharing). A user marks one instance as `isPrimary` per plugin — that's the default for outbound sends when they own the lead.

When a BDR leaves, the admin swaps the `ChannelAccess` and renames the instance. History and credentials stay attached to the instance.

### CRM is a connector, not native

Pipedrive enters as a connector. CRM events (inbound webhook) are normalized into an internal vocabulary (`lead.stage_entered`, etc.). Outbound actions (Activity, stage change, mark lost) go through the connector's API. Cadences consume normalized events — they don't know Pipedrive.

A native CRM (deals, own pipeline) is phase 2.

## Domain

Entities:

- `Workspace`
- `User` — global identity (email, password hash)
- `Membership` — `{ workspaceId, userId, role: 'admin' | 'member', status: 'active' | 'inactive' }`
- `Lead` — `workspaceId: Workspace.id`, `ownerId: User.id` (mirrored from Pipedrive `deal.user_id`)
- `Cadence` — aggregate with steps, policy, hooks
- `Step` — `order`, `delay`, `jitter`, `channelRef`, `template`
- `Template` — message body
- `LeadJourney` — execution state per lead (`running | paused | replied | exhausted | stopped | paused_owner_inactive`)
- `TouchAttempt` — record of each send (success, error, externalActivityId in the CRM)
- `ChannelAccount` — channel plugin instance (no `ownerId`; workspace is the owner)
- `ChannelAccess` — `{ channelAccountId, userId, isPrimary }`
- `EntryTrigger` — list of `{ connectorId, pipelineId, stageId, cadenceId }` configurations

## Plugins and connectors

### Channel plugin (Evolution / WhatsApp)

Loading: module within the monorepo (not an external npm package, not a separate process). Reassess isolation when a third-party, community-contributed plugin shows up.

Contract (sketch):

```
ChannelPlugin {
  manifest: { id, name, capabilities, configSchema }
  send(payload, credentials): SendResult
  parseInbound(raw, credentials): InboundMessage[]
  validate(step, leadState, channelState): Decision
}
```

Capabilities declared statically in the manifest (`['freeform', 'template', 'media']`). `validate` is the executable hook that captures runtime peculiarities (24h window, opt-in, HSM).

### CRM connector (Pipedrive)

Contract (sketch):

```
CRMConnector {
  manifest: { id, name, capabilities, configSchema }
  parseWebhook(raw, config): NormalizedEvent[]
  fetchLead(externalId, credentials): NormalizedLead
  logActivity(externalId, activity, credentials): { externalActivityId }
  moveStage(externalId, stageRef, credentials): void
  markLost(externalId, reason, credentials): void
  setField(externalId, field, value, credentials): void
}
```

Pipedrive specifics:

- Auth via per-workspace API token (OAuth when cloud exists).
- `deal.updated` webhook filtered by `stage_id` transitions (uses `previous` + `current`).
- Activity of configurable type (`task` or custom `whatsapp_message`), with `user_id` = deal owner for correct timeline attribution.
- Native `lost_reason` field (no custom field required).
- Rate limit ~100 req / 10s — outbound queue with throttle.
- `configSchema` allows mapping "which custom field holds the phone" for operations that don't use Pipedrive's default `phone`.
- Webhook idempotency key: `pipedrive:deal:{id}:event:{event}:{timestamp}`.

## Engine

Scheduler:

- Dispatches touches when `LeadJourney.nextTouchAt <= now`.
- Respects the cadence's `sendingWindow` (timezone, days, hours).
- Applies per-step `jitter`.
- Resolves channel: `step.channelRef.strategy = 'lead_owner'` → looks up `ChannelAccess` where `userId = lead.ownerId`, `pluginId = step.pluginId`, `isPrimary = true`.
- If no channel is available: the journey transitions to `error_state` with an explicit reason (never silent).
- Idempotency keyed by `(leadJourneyId, stepOrder)` to prevent double dispatch under race.

Inbound:

- Unique webhook URL per `ChannelAccount` (`/webhook/channel/{id}?secret=...`).
- Plugin parses into `InboundMessage { channelAccountId, fromExternalId, body, ts }`.
- `LeadJourney` lookup by `(lead.phone, channelAccountId)` or similar.
- Reply marks `LeadJourney.status = 'replied'` and triggers `onReply` actions.

Closed-vocabulary actions (not arbitrary code):

- `move_stage(stageRef)` — via CRM connector
- `mark_lost(reason)` — via CRM connector
- `log_activity(activity)` — via CRM connector
- `notify_user(userId)` — internal
- `set_field(key, value)` — on Lead or CRM
- `webhook_out(url, payload)` — single escape hatch

Inactive memberships: journeys where the `Membership` matching `(lead.workspaceId, lead.ownerId)` is inactive move to `paused_owner_inactive`. Admin reassigns leads in bulk (UI or API). No automation in v0.1.

## API

- REST + OpenAPI from day one.
- CRUD for every domain entity.
- Authenticated public endpoint for CRM-side webhook ingestion.
- Authenticated public endpoint for channel inbound (URL per `ChannelAccount`).
- The UI consumes the same API (API-first).

## Minimum UI

Auth: login (email/password or magic link — pick one).

Workspace (admin):

- User management (create, mark inactive, reassign leads).
- Workspace channels (create instance, configure credentials, generate webhook URL, grant access).
- Pipedrive connector (token, pipeline+stage → cadence mapping).

BDR:

- My channels (instances I have access to, set primary).
- Cadences and templates (CRUD).
- Inbox: conversations from instances I have access to, filter by instance and by "my leads".
- List of active/paused/error journeys.

Visual workflow builder UI: out of scope.

## Out of scope for v0.1

Deliberately excluded to avoid scope creep:

- Workflow engine / generic triggers / visual builder
- Conditional branching inside a step
- Automatic channel fallback (if WhatsApp fails, try SMS)
- Multi-channel per step
- Granular RBAC (viewer / sender / admin distinct from membership)
- AI / reply classification / touch generation
- Native CRM (deals, own pipeline)
- Forms / landing pages / scoring / enrichment
- SSO, audit log
- Multi-tenant cloud
- Automatic lead reassignment when a user is deactivated
- Round-robin / load balancing across channels

Every item above is phase 2+. Pulling any of them in now would freeze API decisions too early.

## Auth

Home-grown, minimal: `User` (email, password hash), session table, basic login/logout. No external auth library in v0.1 — avoids coupling an auth library's organization/teams schema with our domain. `Workspace` and `Membership` are domain entities, not auth concerns, and stay 100% in domain tables.

Trade-off accepted: a couple extra weeks of careful work on primitives (password reset, session expiry, CSRF, login rate-limit). Payoff: a future swap of the auth implementation touches only the auth boundary, never `Lead` / `Cadence` / `ChannelAccount` FKs.

Magic link vs. password: pick one for v0.1.

## Open decisions

To be settled before any code:

1. **`LeadJourney` state machine** — transitions, scheduler idempotency, race condition between touch dispatch and inbound reply.
2. **Final channel plugin contract** — exact signature, manifest format, isolation.
3. **Final CRM connector contract** — `NormalizedEvent` shape, pagination, retries.
4. **Persistence** — schema, migrations, ORM/query builder, transactions.
5. **Scheduler** — implementation (in-process cron, external queue, etc.).

## Assumptions to confirm with the pilot

1. Pipedrive pipelines: per-BDR or shared with owner-per-deal? (Model supports both; helps for config UI.)
2. Each BDR has their own Evolution instance (confirmed).
3. Simple auth (no SSO) is acceptable for the pilot.
4. Manual lead reassignment when a BDR leaves is acceptable.
