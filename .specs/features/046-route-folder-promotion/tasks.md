## Execution Plan

All tasks are **sequential** — every task touches `routeTree.gen.ts` (regenerated by
`@tanstack/router-plugin`) and the TypeScript graph, so two `[P]` tasks would race
on the generated file and on imports. The tasks are short and mechanical (file
moves + path rewrites + regenerate), so sequencing is cheap.

```
Phase 0 (graduate cross-feature primitive):
  T1

Phase 1 (add redirects — break the two naked URLs):
  T2 ─→ T3

Phase 2 (promote settings features, lightest to heaviest):
  T4 ─→ T5 ─→ T6 ─→ T7 ─→ T8 ─→ T9 ─→ T10 ─→ T11

Phase 3 (promote workspace features, lightest to heaviest):
  T12 ─→ T13 ─→ T14 ─→ T15 ─→ T16

Phase 4 (codify rules + update memory):
  T17 ─→ T18
```

Per-task gate: **Build (`bun check`)**. Per AGENTS.md DoD, every commit must
leave the gate green; this is a structural refactor, so `bun check` is the
relevant gate for every commit (no DB/HTTP involved, but typecheck + lint +
script-gated checks all matter). No new tests are authored — existing tests
exercise the moved code and must keep passing.

## Notes

- **Why sequential, not parallel?** Every task regenerates `routeTree.gen.ts`
  and modifies imports. Two `[P]` tasks would race on the generated file
  and on cross-file rename consistency. Sequential execution is cheap and
  bisect-friendly (each commit is one feature's promotion).
- **Why `bun check` per task, not just at the end?** AGENTS.md DoD: "never
  commit with a red `bun check`". Each atomic commit is a passing-gate
  commit.
- **Why no new tests?** This is a structural refactor; behavior is
  unchanged. New `index.tsx` redirects are thin orchestration per
  TESTING.md — `beforeLoad` throws a router `redirect()`. A unit test for
  that would either mock `redirect` and assert it was called with a literal
  (restates the code) or assert `beforeLoad` throws (restates the code).
  Covered by P1 acceptance criteria (manual smoke + visual confirmation
  that `/auth` and `/settings` redirect).
